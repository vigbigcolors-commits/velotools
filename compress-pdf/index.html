Technical deep-dive

How the compression actually works

Unlike server-based tools that are a black box, our compression pipeline uses three open-source libraries that you can audit. Here’s what each one does — and why this approach is both more private and more transparent than cloud alternatives.

PDF Parsing

PDF.js

Mozilla’s open-source PDF rendering engine, used in Firefox’s built-in PDF viewer. It parses the raw PDF binary format, resolves cross-reference tables, and renders each page to an HTML Canvas element at the specified DPI. Every page becomes a bitmap canvas at exactly the resolution you chose. The canvas is then compressed as JPEG, which is where the size reduction happens. PDF.js handles complex PDFs including multi-layer, form-containing, and encrypted documents.

PDF Assembly

pdf-lib

A JavaScript library for creating and modifying PDF documents from scratch. After PDF.js renders each page as a JPEG image, pdf-lib assembles these compressed images into a brand new PDF document. The output is a clean, minimal PDF containing only the compressed page images — no embedded fonts, no metadata, no form fields, no annotations (unless you chose to keep them). This clean construction approach produces smaller files than attempting to strip content from the original PDF in-place.

ZIP Packaging

JSZip

A client-side ZIP file creation library used for the “Download All (ZIP)” batch download feature. When you compress multiple PDFs and click to download all, JSZip collects all compressed PDF bytes in browser memory and packages them into a single .zip archive — entirely locally. No server is involved in the ZIP creation. The ZIP is assembled on your device and downloaded directly. JSZip also powers the per-file naming (preserving the original filename with _compressed suffix).

01

FileReader API loads PDF bytes into browser memory

The PDF is read by the browser’s native FileReader API into an ArrayBuffer. This is a direct local memory operation — no network connection is involved. The raw bytes live in your browser tab’s memory and are never transmitted anywhere.

02

PDF.js parses and renders each page to a canvas at your chosen DPI

PDF.js receives the array buffer and parses the PDF structure. Each page is rendered onto an opaque white canvas at scale = DPI / 72. Grayscale is applied after render via a safe compositing pass (never raw pixel overwrite, which breaks transparency). Annotations are skipped when “Remove annotations” is enabled.

03

Canvas is converted to JPEG bytes via toDataURL

The rendered canvas is encoded as a JPEG data URL at your quality setting (0.1–0.95). This is where the majority of compression occurs: JPEG’s DCT (Discrete Cosine Transform) algorithm is extremely efficient at compressing photographic and scanned content. A canvas image at 40% quality retains full perceptual quality at normal screen zoom while being ~10× smaller than a lossless PNG.

04

pdf-lib assembles all pages into a new clean PDF

Each page’s JPEG bytes are embedded as images in a new PDF document created from scratch by pdf-lib. The output PDF uses Object Streams compression internally, and includes only the page images — no embedded fonts, no JavaScript, no metadata (unless “Strip metadata” is off). The final output is downloaded as {filename}_compressed.pdf.

Real use cases

Who compresses PDFs every day

PDF compression is one of the most universal document tasks — it spans every industry that handles files digitally. Here’s how six distinct professional contexts use it, and what each one specifically needs from a compression tool.

⚖

Legal Professionals

Law firms handle vast volumes of scanned court filings, contracts, discovery documents, and case files. These are almost always scanned PDFs — the highest-compression use case. A court filing that starts at 80MB can reach 8MB at Web preset with no practical quality loss. Confidentiality is paramount: uploading client documents to any third-party server violates legal privilege in most jurisdictions. Browser-based compression solves this cleanly. Use Print preset for documents that will be filed physically.

📈

Marketing & Agencies

Marketing agencies produce large presentation PDFs, brand books, media kits, and proposal decks filled with high-resolution images. These routinely exceed 50–100MB in their native form. Web preset reduces a 60MB brand guide to 18MB — small enough for client email while preserving the visual quality that justifies the creative work. Batch mode is essential for agencies: compress the full proposal deck, the rate card, the case studies, and the appendices simultaneously before a client meeting.

🏫

Students & Academics

University online portals and learning management systems (Blackboard, Canvas, Moodle) routinely enforce 5MB or 10MB upload limits for assignment submissions. A thesis with scanned references or a research report with figures easily exceeds these limits. Screen or Web preset reduces most student PDFs to well within portal limits. Academic institutions also frequently require scanned physical forms — admission documents, transcripts, letters of recommendation — that compress dramatically from their raw scan sizes.

💼

HR & Operations

Human resources departments handle continuous flows of CVs, onboarding packets, signed offer letters, tax forms, ID copies, and policy acknowledgements. Scanned HR documents compress 70–85% with the Web preset. The grayscale option is particularly effective for black-and-white forms and letterheaded documents — most HR paperwork has no colour content worth preserving. Batch compression of incoming CV pools (20 at once) dramatically reduces storage requirements on shared drives and HR systems.

⚕

Healthcare Administration

Medical records, patient intake forms, insurance claims, and referral letters are frequently scanned physical documents. These are among the most sensitive files that exist — sharing them with a third-party cloud service to compress them is a clear HIPAA violation in US healthcare contexts and a GDPR violation in European contexts. Browser-based compression is not just convenient here; it’s the only architecturally compliant approach. Archive preset is recommended to maintain diagnostic-quality image content in clinical photographs.

🏠

Real Estate

Property listings, valuation reports, survey documents, lease agreements, and planning permissions are all PDF-heavy workflows. A 40-page survey report with site photographs often starts at 35–60MB. Web preset reduces this to 10–18MB for portal upload and client email. Lease agreements and contracts are better served by Print preset to maintain quality for signing and physical filing. Real estate portals (Rightmove, Zillow, Zoopla) all have document size limits that batch PDF compression routinely solves.

Expert tips

Get maximum compression without surprises

📷

Scanned PDFs compress 70–90%

Scanned documents are images inside a PDF wrapper — every page is already raster data. JPEG compression on raster content is extremely efficient. A 20-page scanned contract at 15MB will typically reach 1.5–3MB at Screen or Web preset. This is the compression scenario the tool is optimised for. If your PDF looks like a photo of a page, expect dramatic results.

📋

Enable grayscale for monochrome documents

If your document is black and white — contracts, letters, financial statements, academic papers — enabling Grayscale mode removes all colour channel data. JPEG compresses single-channel grayscale images significantly more efficiently than three-channel colour images. Expect an additional 15–25% size reduction on top of the JPEG preset savings, with zero visible quality difference on monochrome content.

💾

Text PDFs compress less — and that’s normal

A PDF created in Microsoft Word, Google Docs, or InDesign with clean vector text compresses 10–25% rather than 70–90%. This is not a tool limitation — it’s the nature of the content. Clean vector text, when rendered to canvas at any DPI, produces very JPEG-unfriendly image data (sharp edges, high contrast). The main savings for text PDFs come from metadata removal and re-encoding. For text PDFs, Archive preset is often the best choice to avoid any visible degradation.

🔌

Use ZIP download for batch workflows

When processing multiple files, avoid clicking individual download buttons — click “Download All (ZIP)” once all files are done. The ZIP is assembled entirely in your browser (JSZip) with no server involvement. Files retain their original names with _compressed suffix for easy identification. The ZIP approach is also cleaner for sharing: a single compressed archive is easier to attach to an email or upload to a shared drive than 15 individual files.

FAQ

Frequently asked questions about PDF compression

Does this PDF compressor upload my files to a server?
Absolutely not. Every step happens in your browser using JavaScript. Your PDF file never leaves your device. You can turn off your internet connection after the page loads and compression still works perfectly. The three libraries (PDF.js, pdf-lib, JSZip) run entirely client-side. There is no API call, no presigned upload URL, no S3 bucket — the compression architecture makes server transmission structurally impossible, not just policy-prohibited.
How much can I reduce a PDF file size?
It depends entirely on the content type. Scanned documents (images of physical pages): 70–90% smaller with Screen or Web preset. Presentation PDFs with photos and graphics: 50–80% smaller. Text-only PDFs (Word exports, contracts, academic papers): 10–25% smaller, primarily from metadata removal and re-encoding inefficiencies. The before/after size display in the tool shows your exact result in real time.
Will compression reduce the visual quality of my PDF?
Yes — every preset except Archive involves a quality trade-off. Archive (JPEG 90%): virtually indistinguishable from original at any zoom. Print (JPEG 80%): essentially invisible quality difference at reading scale. Web (JPEG 65%): slight softening visible only at 300%+ zoom. Screen (JPEG 40%): visible artifacts at close zoom, completely acceptable at normal reading scale. For final client deliverables or documents that must survive close inspection, use Print or Archive. For internal sharing and email, Web is optimal.
What is the difference between DPI and JPEG quality?
These are two independent compression levers that multiply together. DPI (dots per inch) determines the canvas resolution each page is rendered at before encoding. Lower DPI = smaller canvas = smaller source image before compression. 72 DPI is screen resolution; 300 DPI is print quality. JPEG quality (10–95%) controls how aggressively the JPEG encoder compresses the canvas image. Both controls affect file size: halving the DPI reduces the image area by 75%; reducing quality from 90% to 40% reduces JPEG size by roughly 60%. The presets combine both levers for sensible defaults.
Is there a file size or page count limit?
No hard limits — the constraint is your device’s available RAM. Each page requires memory proportional to DPI²: a 300 DPI canvas for an A4 page is approximately 3500×4950 pixels, requiring ~66MB of uncompressed bitmap memory per page. At 96 DPI, the same page requires ~6MB. For very large PDFs (200+ pages) or very high DPI settings, compression may slow significantly or trigger memory warnings on devices with limited RAM. Most 100-page scanned PDFs at Web preset process without issues on a modern laptop or desktop.
Can I compress multiple PDFs at once?
Yes — up to 20 PDFs simultaneously. Drag and drop them all at once or use the file picker with multi-select. Files are processed sequentially (one at a time internally) to prevent memory spikes, but each shows a real-time progress bar. As each file finishes, its download button appears immediately. When all files are done, “Download All (ZIP)” becomes available — it packages all compressed files into a single ZIP archive assembled in your browser via JSZip.
What does “Strip metadata” do?
When enabled (default), the output PDF has author, creation date, title, and producer fields cleared. This removes identifying information and saves a small amount of space. Turn it off if you need to preserve the original document metadata.
Why does my text-only PDF barely compress?
Our compressor works by re-rendering pages as JPEG images. Text rendered to a canvas at any DPI produces hard-edged, high-contrast pixel data — exactly the content type JPEG handles least efficiently (JPEG is designed for photographic content with smooth gradients). A Word-export PDF with clean text at Web preset might only compress 15–20%, compared to 70–80% for the same page count with scanned content. For text-heavy PDFs, Archive preset with grayscale enabled is usually the right choice — it minimises quality loss while still achieving meaningful savings from re-encoding.
Can I use this for confidential or legally sensitive documents?
Yes — this is specifically one of the most appropriate use cases. Because the compression happens entirely in your browser with no network transmission, there is no data controller relationship, no third-party data processor agreement needed, and no risk of document exposure through a third-party security breach. For law firms (legal privilege), healthcare providers (HIPAA/GDPR), financial institutions (data protection regulations), and anyone handling NDA-covered content, browser-based compression is the architecturally correct approach.
Does it work offline?
Yes, fully. After the page loads and the three JavaScript libraries (PDF.js ~2MB, pdf-lib ~0.8MB, JSZip ~0.3MB) are cached in your browser, the tool works without any internet connection. Go to airplane mode — compression, batch processing, and ZIP download all continue to work perfectly. This is useful for compressing documents on trains, planes, or in environments with restricted internet access. The browser cache persists until you clear your browser data.
